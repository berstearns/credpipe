# pane: `serve` (window 3) — KEY + .env DISTRIBUTION

Role: serve the source-of-truth (`.env` + the passphrase-wrapped key) to a tty node over
do-automation/serve (HTTPS + basicauth + IP-allowlist + fail2ban). Session `credpipe-demo`,
pane title `cp-serve`, cwd `<repo>`.

> Real per-deployment values are masked as `<…>`. Unmasked copy: git-ignored `runbook/`.

## Command run (final form)

```sh
cd <repo> && CREDPIPE_KEY_PASSPHRASE='<passphrase>' bash setup/serve-key-session.sh
```

- `CREDPIPE_KEY_PASSPHRASE` — **you choose this**; it wraps the key for transport. Must match
  the value used on the tty node.
- The script brings the serve up and **prints the passcode** in its log output:
  `passcode: <passcode>`  ← copy this into the tty-node command.
- Serves two slots: `/i.sh` = the `.env` (config), `/r.conf` = `key.enc` (the wrapped key).
- Locks the serve port (47823) to this host's public IP.
- Then **blocks**. Press **Ctrl-C** to tear down (caddy off, key purged, bootstrap firewall
  deleted) — that's what `serve-key-session.sh` adds over plain `serve-key.sh`.

## Manual equivalents

```sh
CREDPIPE_KEY_PASSPHRASE='<passphrase>' bash setup/serve-key.sh   # up, then exits (stays serving)
bash setup/serve-key-down.sh                                     # explicit teardown
bash setup/serve-key-purge.sh                                    # robust purge (do-serve-down self-kills)
```

## Notes

- The **passcode** is generated by the serve and shown in the logs — the one value you carry to
  the tty node. The **passphrase** is chosen by you (same on both sides).
- The real shared key (`~/.config/credpipe/key`) never leaves here unwrapped; only `key.enc`
  (passphrase-encrypted) is served.
